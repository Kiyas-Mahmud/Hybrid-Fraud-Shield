                    REQUIREMENTS ANALYSIS (From Image)
================================================================================

USER FLOW REQUIREMENTS:
-----------------------

1. User Login
   → Mobile data and WiFi must be connected
   → Authentication required

2. Transaction Initiation
   → User tries to make a transaction
   → All data collected and split into 72 features

3. API Integration
   → API catches transaction with classification: Fraud, Suspicious, Safe
   → All details/results should be shown in admin panel

4. User Notification Flow (IN-APP ONLY - NO EMAIL/SMS)
   → User sees IN-APP NOTIFICATION PAGE immediately
   → Page shows transaction details and risk assessment
   → Message: "Transaction requires verification. Please review details below."
   → Options:
      - YES (Approve) → Transaction proceeds
      - NO (Reject) → Transaction blocked permanently
                       Account flagged for admin review
                       Admin dashboard updated in real-time

5. Risk-Based Routing
   → If SUSPICIOUS → Some additional verification
   → If SAFE → Go to next step

MILESTONE 1.1: CORE API ENHANCEMENTS (Week 1)
---------------------------------------------------------------------------

Task 1.1.1: Update Existing API to Support 72 Features
□ Status: Current API expects 63 features, need to update to 72
□ Files to modify:
  - Fusion_API/app/main.py (TransactionRequest model)
  - Fusion_API/app/preprocessing.py (feature validation)
  - Fusion_API/app/inference.py (model input handling)
□ Deliverable: Updated API accepting 72 features

Task 1.1.2: Implement Risk Classification Endpoint
□ Create new endpoint: POST /api/v1/classify
□ Risk Categories:
  - SAFE: probability < 0.4
  - SUSPICIOUS: 0.4 <= probability < 0.7
  - FRAUD: probability >= 0.7
□ Response format:
  {
    "transaction_id": "txn_123",
    "classification": "SUSPICIOUS",
    "confidence": 0.65,
    "risk_score": 65,
    "timestamp": "2025-11-08T12:34:56Z"
  }
□ Files to create/modify:
  - Fusion_API/app/risk_classifier.py (new)
  - Fusion_API/app/main.py (add endpoint)

Task 1.1.3: Add User Authentication System
□ Implement JWT-based authentication
□ Endpoints:
  - POST /api/v1/auth/login
  - POST /api/v1/auth/logout
  - POST /api/v1/auth/refresh
  - GET /api/v1/auth/verify
□ User roles:
  - USER: Can submit transactions
  - ADMIN: Can view dashboard, approve/reject transactions
  - SUPER_ADMIN: Full system access
□ Files to create:
  - Fusion_API/app/auth.py (new)
  - Fusion_API/app/database.py (user management)
  - Fusion_API/app/models.py (User, Session models)
□ Database: SQLite (dev) / PostgreSQL (production)

MILESTONE 1.3: USER RESPONSE HANDLING (Week 2)
---------------------------------------------------------------------------

Task 1.3.1: Verification Endpoint
□ Create endpoint: POST /api/v1/verify-transaction
□ Request:
  {
    "transaction_id": "txn_123",
    "verification_code": "ABC123",
    "user_response": "YES" | "NO"
  }
□ Response:
  {
    "status": "approved" | "rejected" | "blocked",
    "message": "Transaction approved",
    "next_action": "proceed" | "contact_admin" | "blocked"
  }
□ Files to create:
  - Fusion_API/app/verification.py

Task 1.3.2: Account Blocking Logic
□ Implement temporary and permanent blocking
□ Features:
  - Auto-block on "NO" response
  - Notify admin via dashboard
  - Send confirmation to user
  - Log all actions for audit
□ Files to modify:
  - Fusion_API/app/database.py (add blocking tables)
  - Fusion_API/app/auth.py (check blocked status)

Task 1.4.1: Transaction Monitoring Endpoints

  - Reason required

  - Fraud trends over time

□ GET /api/v1/admin/analytics/models

  - Meta-learner statistics
  - SHAP importance rankings

Task 1.4.3: User Management Endpoints
□ GET /api/v1/admin/users

□ GET /api/v1/admin/users/{user_id}

□ POST /api/v1/admin/users/{user_id}/block
  - Block user account
  - Reason required

□ POST /api/v1/admin/users/{user_id}/unblock
  - Unblock user account
  - Admin approval required
